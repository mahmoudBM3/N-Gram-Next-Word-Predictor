1. Overview

This capstone project consolidates the Python programming, software engineering, and AI fundamentals covered in the course. You will build a next-word prediction system from scratch using an n-gram language model — a statistical approach to AI.

The project is structured as four core modules plus one optional extra credit module: data preparation, model training, inference, and a CLI. A Streamlit UI is available as an optional extra credit module.

The corpus is four Sherlock Holmes novels by Arthur Conan Doyle, sourced from Project Gutenberg. The model is trained on all four novels. A fifth novel is used as a held-out evaluation corpus by the Model Evaluator extra credit module. At inference time, the system takes the last NGRAM_ORDER − 1 words typed by the user and returns the top-k most probable next words using an n-gram model with backoff to lower orders when the context is unseen. The primary deliverable is a CLI; a Streamlit UI is available as extra credit.

The aim is not to test you, but to build on your Python training and teach you how to develop a real-world programming project. Professional habits — tooling, code structure, documentation, and testing — matter as much as algorithmic correctness, and this project enforces both.

Reference: Chen, S. F., & Goodman, J. (1999). An empirical study of smoothing techniques for language modeling. Computer Speech & Language, 13(4), 359–394. https://people.eecs.berkeley.edu/~klein/cs294-5/chen_goodman.pdf

4. General Coding Instructions

4.1 Setup

  • Apply tooling, code structure, and standard parts from day one. The VS Code environment, Git workflow, folder structure (§5), and standard files — README.md, requirements.txt, .gitignore, and config/.env — must be in place before writing any code, not added at the end.
  • Git workflow. The first commit must be the empty project structure — folders, placeholder files, and all standard parts — before any code is written. Commit regularly as you complete each step, not all at once at the end. Follow the milestone branch structure in §7.
  • Load environment variables at startup. Use the python-dotenv library. Call load_dotenv() at the top of main.py before reading any path or configuration value. All other modules receive config values as arguments — they do not call load_dotenv() themselves.
  • Every module file must have a runnable entry point. Each .py file in src/ must include a main() function and an if __name__ == "__main__" block so the module can be run and tested in isolation during development.
  • Build on a small sample first. During development, run all modules on the first 100 sentences of the training corpus only. Once the full pipeline works end-to-end on the sample, re-run on the full corpus. This keeps iteration fast and avoids long waits while debugging.
  • Do not use AI-assisted programming tools (e.g. GitHub Copilot, ChatGPT, Claude) anywhere in the core modules. AI tools are permitted for extra credit modules only.

4.2 Design

  • Docstrings on all classes and methods. Every class docstring must describe the module's responsibility. Every method docstring must describe what the method does, its parameters, and its return value.
  • No global variables. All state is passed via function arguments or class attributes.
  • No code duplication. Each piece of logic must exist in exactly one place. If two modules need the same behavior, one calls the other — never copy code between files.
  • Normalization must be consistent. Any text processed at inference time must pass through Normalizer.normalize() in the same order as training data — otherwise lookups will silently fail. Do not re-implement normalization in any other module. Note: Normalizer has two modes of use — in Module 1 it processes whole files; in Module 3 it normalizes single input strings. The same methods are called in both cases.
  • Dependency injection. NGramModel and Normalizer are instantiated once in main() and passed into Predictor. No class creates its own dependencies internally. See Appendix A for a full explanation.

6.2 Module 2 — Model

IN train_tokens.txt
OUT model.json, vocab.json

Reference Paper

Read Section 2 (MLE baseline) and Section 3.4 (Katz Backoff) of the reference paper to understand the motivation for building counts at multiple n-gram orders and falling back to lower-order models when a context is unseen.

The approach implemented here is a simplified backoff — no discounting or probability redistribution as in Section 3.4. Do not implement any full smoothing techniques from Sections 3–6. Smoothing is offered as an optional extension for extra credit.

Note: This is a prototype, not a production system. The goal is a correctly functioning model with backoff, not an optimized one. The n-gram order is set by NGRAM_ORDER in config/.env (default: 4). Build and test the counts and backoff logic to support any order from 1 up to NGRAM_ORDER — do not hardcode 4 anywhere in the code.
Code hint — backoff loop: The lookup logic iterates from the highest order down to 1-gram. The shape of the loop is: 
for order in range(ngram_order, 0, -1):
    context = words[-order + 1:]   # shrinks with each step
    ...
Do not copy this verbatim — understand what each line does and adapt it to your implementation.

Algorithm — MLE with Stupid Backoff

Build MLE counts at all n-gram orders from 1 up to NGRAM_ORDER (read from config/.env). At inference time, attempt the highest-order match first and fall back to lower orders if the context is unseen. The number of backoff levels is determined by NGRAM_ORDER — do not hardcode any specific order value in the code.

Let n = NGRAM_ORDER (e.g. 4) Try n-gram: P(w | w₋ₙ₊₁…w₋₁) = C(w₋ₙ₊₁…w₋₁ w) / C(w₋ₙ₊₁…w₋₁) → if context not seen, fall back to: Try (n-1)-gram: P(w | w₋ₙ₊₂…w₋₁) = C(w₋ₙ₊₂…w₋₁ w) / C(w₋ₙ₊₂…w₋₁) → if context not seen, fall back to: ... Try 2-gram: P(w | w₋₁) = C(w₋₁ w) / C(w₋₁) → if context not seen, fall back to: Try 1-gram: P(w) = C(w) / C(all words) Return candidates from the first successful lookup.

Steps

01

Build the vocabulary — collect all unique words; replace any word appearing fewer than UNK_THRESHOLD times (from config/.env) with <UNK>; add <UNK> to the vocabulary.

02

Build counts at all orders — slide a window across every sentence and count all unique n-grams from 1-gram up to NGRAM_ORDER-gram. The number of orders is read from config/.env.

03

Compute MLE probabilities at each order — for each n-gram, divide its count by the count of its (n-1)-gram prefix. For unigrams, divide by the total word count.

04

Handle out-of-vocabulary words — any word in the context not found in the vocabulary must be mapped to <UNK> before lookup.

05

Save the model — save all probability tables as a single model.json with one key per order (e.g. "1gram" through "4gram" when NGRAM_ORDER=4).

Output Format

The example below shows NGRAM_ORDER=4. The number of keys in model.json equals NGRAM_ORDER.

// vocab.json
["the", "holmes", "said", "watson", "<UNK>", "..."]

// model.json (example: NGRAM_ORDER=4)
{
  "1gram": {"the": 0.05, "holmes": 0.03, "<UNK>": 0.01},
  "2gram": {"holmes said": {"watson": 0.42, "the": 0.12}},
  "3gram": {"holmes said to": {"watson": 0.38, "me": 0.21}},
  "4gram": {"holmes said to watson": {"i": 0.31, "and": 0.18}}
}

The class docstring must describe the module's responsibility: building, storing, and exposing n-gram probability tables and backoff lookup across all orders from 1 up to NGRAM_ORDER. Each method docstring must describe what that method does, its parameters, and its return value.

6.3 Module 3 — Inference

IN NGramModel instance, Normalizer instance, text string
OUT top-k words sorted by probability

Algorithm — Backoff Lookup

Let n = NGRAM_ORDER (e.g. 4) Given input text, normalize it and extract the last n−1 words as context. Try n-gram: look up context of length n−1 — highest order match → if found, rank candidates by probability Try (n-1)-gram: look up context of length n−2 → if found, rank candidates by probability ... Try 2-gram: look up context of length 1 → if found, rank candidates by probability Try 1-gram: look up all words by unigram probability Return top-k words from the first successful lookup.

Steps

01

Normalize input — call Normalizer.normalize(text) on the user's input string. Extract the last NGRAM_ORDER − 1 words as context. Do not reload the model here — it is passed in via the constructor.

02

Map OOV words — any word in the context not found in the vocabulary is replaced with <UNK> before lookup.

03

Backoff lookup — call NGramModel.lookup(context). The model tries the highest-order context first and falls back to lower orders automatically. Do not re-implement backoff here.

04

Rank and return — sort candidates by probability (highest first); return the top-k words, where k is read from TOP_K in config/.env. If no predictions are found at any order, return an empty list.

Class Structure

Implement a Predictor class in src/inference/predictor.py. The class docstring must describe the module's responsibility: accepting a pre-loaded NGramModel and Normalizer via the constructor, normalizing input text, and returning the top-k predicted next words sorted by probability. Backoff lookup is delegated to NGramModel.lookup(). Each method docstring must describe what that method does, its parameters, and its return value.

MethodResponsibility
__init__(model, normalizer)Accept a pre-loaded NGramModel and Normalizer instance. Do not load files here.
normalize(text)Call Normalizer.normalize(text); extract last NGRAM_ORDER − 1 words as context
map_oov(context)Replace out-of-vocabulary words with <UNK>
predict_next(text, k)Orchestrate normalize → map_oov → NGramModel.lookup() → return top-k words sorted by probability

Output Format

["watson", "holmes", "inspector"]

6.4 Module 4 — CLI

IN Predictor instance, text string
OUT top-k predictions printed to terminal

The primary deliverable is a command-line interface. The user types a sequence of words and the program prints the top-k next-word predictions. The CLI is implemented in main.py at the root of the repository — no separate file is needed.

main.py Spec

main.py accepts a --step argument that controls which part of the pipeline to run:

CommandWhat it runs
python main.py --step dataprepRun Normalizer — produce train_tokens.txt
python main.py --step modelRun NGramModel — produce model.json and vocab.json
python main.py --step inferenceStart the interactive CLI prediction loop
python main.py --step allRun dataprep → model → inference in sequence

Parse --step using Python's built-in argparse module. load_dotenv() must be called before any step runs.

Behavior

  • On startup, load config/.env, instantiate Normalizer and NGramModel, load the model, and instantiate Predictor.
  • Enter a loop: prompt the user for input, call Predictor.predict_next(text, k), and print the top-k predictions.
  • Exit cleanly on quit or Ctrl+C.

Example Session

9.5 Model Evaluator

+5 points. Implement the Evaluator class in src/evaluation/evaluator.py to compute perplexity on the held-out evaluation corpus. Download The Valley of Fear (Gutenberg ID 3289) and place it in data/raw/eval/ (see §2 for the download link).

Read Section 2.1 of the reference paper for the cross-entropy definition. Perplexity is 2 raised to the power of cross-entropy.

For each word wᵢ in the eval corpus: Get P(wᵢ | context) via NGramModel.lookup() with backoff Accumulate log₂ P(wᵢ | context) Cross-entropy H = - (1/N) × Σ log₂ P(wᵢ | context) Perplexity = 2^H Where N = total number of evaluated words (skipped words excluded).

A — Dependency Injection

This project uses a design pattern called dependency injection. Instead of each class creating the objects it needs internally, those objects are created once externally and passed in at construction time.

Without dependency injection

class Predictor:
    def __init__(self):
        self.model = NGramModel()        # Predictor creates its own model
        self.normalizer = Normalizer()   # and its own normalizer

class PredictorUI:
    def __init__(self):
        self.predictor = Predictor()     # PredictorUI creates its own Predictor

This creates multiple instances of the same objects, wastes memory, and makes it impossible to share state between modules.

With dependency injection

model      = NGramModel()
model.load("model.json", "vocab.json")   # Loaded once in main()

normalizer = Normalizer()

predictor  = Predictor(model, normalizer)   # Passed in
ui         = PredictorUI(predictor)         # Passed in

The same model and normalizer objects are shared. No duplication, no inconsistency, and easy to test — pass a small known model in tests without changing any class internals.

Benefits

  • Single instance — the model is loaded once and reused everywhere.
  • No hidden dependencies — every class declares what it needs at construction; nothing is created in secret.
  • Testability — you can inject a small, known model in your tests without touching file paths.
  • Separation of concerns — classes do not decide how their dependencies are built, only how to use them.

In this project, main() is responsible for wiring everything together. Each class is responsible only for its own logic.

B — Unit Testing

A unit test is a small, isolated piece of code that verifies that one method or function does exactly what it is supposed to do. You call the method with a known input and assert that the output matches what you expect. If it does not, the test fails and tells you exactly where the problem is.

Why it matters

  • Catches bugs early. A bug found in a 10-line method is far easier to fix than one discovered when the full pipeline crashes at step 4.
  • Makes refactoring safe. When you change how a method works internally, the tests tell you immediately whether the behavior changed.
  • Documents behavior. A test is executable documentation — it shows exactly what a method should return for a given input.
  • Prevents regressions. Once a bug is fixed, a test for it ensures it never comes back.

Simplest implementation with pytest

Install pytest: pip install pytest. Write a file named test_*.py. Each test is a function whose name starts with test_. Use assert to check the result.

from src.data_prep.normalizer import Normalizer

def test_normalize_lowercases():
    n = Normalizer()
    assert n.normalize("HELLO WORLD") == "hello world"

def test_normalize_removes_punctuation():
    n = Normalizer()
    assert n.normalize("hello, world!") == "hello world"

def test_normalize_removes_numbers():
    n = Normalizer()
    assert n.normalize("chapter 3") == "chapter"

Run all tests from the project root:

pytest tests/

pytest prints a green dot for each passing test and a red F with a description for each failure. You do not need to write a main() or import anything special — pytest finds and runs all test_*.py files automatically.

Key principle

One test, one assertion. Test one behavior per function. If a test fails, you want to know exactly which behavior broke — not have to debug a multi-assertion test to find out.

C — Exception Handling

When something goes wrong at runtime — a file is missing, a key is not in a dictionary, a JSON file is malformed — Python raises an exception. If the exception is not caught, the program crashes and prints a traceback that is often confusing for the user. Exception handling lets you intercept the error, print a useful message, and exit cleanly.

Why it matters

  • Better error messages. A raw Python traceback tells you where the code broke, not why the user's situation caused it. A caught exception can say: "model.json not found — run the Model step first."
  • Graceful failure. Instead of dumping a 20-line traceback, the program exits with a single clear sentence.
  • Easier debugging. When you know exactly which operation failed and why, you fix it faster.

Simplest implementation

Wrap the operation that might fail in a try block. Catch the specific exception type. Print a message that tells the user what went wrong and how to fix it.

import json

# Bad — bare except catches everything, hides the real error
try:
    with open(path) as f:
        model = json.load(f)
except:
    print("Something went wrong")

# Good — catch the specific exception, give a useful message
try:
    with open(path) as f:
        model = json.load(f)
except FileNotFoundError:
    print(f"model.json not found at {path}. Run --step model first.")
except json.JSONDecodeError:
    print(f"model.json is malformed. Delete it and re-run --step model.")

Key rules

  • Never use bare except: — always name the exception type.
  • Catch exceptions as close to the source as possible, not at the top level of main() for everything.
  • The error message must answer two questions: what failed and what should the user do about it.
  • Use sys.exit(1) after printing the message if the program cannot continue.

Common exception types in this project

ExceptionWhen it occurs
FileNotFoundErrorA file or folder path does not exist
json.JSONDecodeErrorA JSON file is empty or malformed
KeyErrorA required key is missing from a dict or .env
ValueErrorA value has the wrong type or is out of range

D — Logging

Logging is the practice of recording what your program is doing as it runs. Instead of using print() statements scattered throughout your code, you use Python's built-in logging module, which lets you control what gets printed, when, and where — without changing the code.

Why it matters

  • Levels of detail. You can turn verbose debug output on during development and off in production with a single config change — no deleting or commenting out print statements.
  • Consistent format. Log messages include a timestamp and severity level automatically, making it easy to trace what happened and in what order.
  • Separation of concerns. Business logic and diagnostic output are separate. Your methods do their work; the logger reports on it.
  • Industry standard. Every production Python system uses logging. Learning it now builds a real professional habit.

Log levels

LevelUse for
DEBUGDetailed internal state — loop iterations, intermediate values. Only useful while debugging.
INFONormal milestones — module started, file loaded, vocab built. Always useful.
WARNINGSomething unexpected but recoverable — OOV word encountered, more than 20% of words skipped.
ERRORSomething failed — caught exception before exiting.

Simplest implementation

import logging
import os

# Configure once at the top of main.py, after load_dotenv()
logging.basicConfig(
    level=os.getenv("LOG_LEVEL", "INFO").upper(),
    format="%(asctime)s [%(levelname)s] %(message)s"
)
logger = logging.getLogger(__name__)

# Then use it anywhere in the same file or module
logger.info("Building vocabulary...")
logger.debug(f"Token count: {len(tokens)}")
logger.warning(f"OOV word encountered: {word}")
logger.error(f"Failed to load model: {e}")

In other modules (normalizer.py, ngram_model.py, etc.), create a module-level logger the same way and use it instead of print():

import logging
logger = logging.getLogger(__name__)

class NGramModel:
    def build_vocab(self, token_file):
        logger.info("Building vocabulary from %s", token_file)
        ...

Key principle

Configure logging once in main(). Every other module just calls logging.getLogger(__name__) and uses it. The configuration — level, format, output destination — is controlled in one place.
